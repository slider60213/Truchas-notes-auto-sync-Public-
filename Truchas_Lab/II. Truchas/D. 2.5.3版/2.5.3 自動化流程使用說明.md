# 🦖 以前

完成 inp 檔設置後：

 1. 輸入模擬指令。
 2. 依照 Output_dt 將不同時間步的結果記錄成 bin 檔。
 3. bin 檔可以透過轉檔指令 轉換成 gmv/tecplot/ensight 檔。
 4. 模擬終止後，若要接續模擬需要先透過轉檔指令生成 restart 檔。
 5. 執行 Restart 前需要先將 Output 資料夾改名避免檔案覆寫。
 6. 輸入 Restart 的指令。
 7. 重複步驟2.、3 來得到新的模擬成果。
 8. 將 gmv/tecplot/ensight 檔移到本地電腦。
 9. 由於這些檔案在 Restart 後也是從 0 開始編號，需要先批量改檔名。
10. 通常會有 4～5個 Output 資料夾的 gmv 檔要重新編號，千萬不要改錯。
11. 完成後才可以用 Matlab 或 Python 讀取。
12. 若模擬途中為了確認成果先轉檔輸出，模擬完成後需要指定輸出的區段或是直接重複輸出。
13. 若是在 Restart 接續模擬途中提前轉檔，還需要再重新編號。

# 👨‍💻 以後

完成 inp 檔設置後：

1. 輸入 `./MakeProblemAuto.sh`。
2. 依照錯誤訊息修正指令為 `./MakeProblemAuto.sh 1 ABCD.inp -gmv`。
3. 直接拉到本地電腦用 Matlab 或 Python 讀取。

# 👨‍🔧 疑難雜症

## 更改資料夾名稱，執行 make clean 後編譯失敗

修改 Python 腳本觸發了 `make` 的全盤重新編譯

你替換了 `TBrookParser` 裡面的 Python 檔案。

- 從頂層 Makefile 的內容可以看到：

makefile

```
TESTMAKE = python runRegressionTests.py -o "$(RESULTSDIR)" -c $(compiler)
```

請謹慎使用程式碼。

- 雖然 Python 是執行期（Runtime）才跑的，但如果 `make` 在掃描目錄時發現有檔案的時間戳記（Timestamp）更新了，或者因為複製檔案時權限變更（你改了 777），可能會導致 `make` 判斷「整個專案的某些部分必須重新編譯」。這就引發了我們前面提到的：**編譯順序被重新洗牌，進而撞上了隨機的循環相依錯誤**。
統一所有人刷新時間（修復 777 帶來的時間混亂）

為了防止有檔案的時間戳記跑到未來去，請在專案根目錄（`truchas-2.5.3`）執行這行指令，把所有檔案的修改時間通通同步成「現在」：

bash

```
find . -type f -exec touch {} +
```

請謹慎使用程式碼。

步驟 3：在 GNUmakefile.sub 加上保險

為了防止多核心平行編譯再次洗牌順序，請打開 `build/` 重新生成後的 `GNUmakefile.sub`（如果 `build/` 被清空了，可以先跑一次 `./Parallel.sh` 讓它生成目錄，或者直接去修改 `src/` 附近的 Makefile 樣板），在檔案最末端補上這行保險：

makefile

```
diagnostics_module.o: time_step_module.o property_module.o
```
