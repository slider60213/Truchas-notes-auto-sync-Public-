這算是錯誤嗎 make[2]: Circular diagnostics_module.o <- property_module.o dependency dropped. make[2]: Circular diagnostics_module.o <- time_step_module.o dependency dropped. make[2]: Circular nucleation_and_growth.o <- time_step_module.o dependency dropped. make[2]: Circular output_utilities.o <- time_step_module.o dependency dropped. make[2]: Circular tbrook_utility.o <- property_module.o dependency dropped. make[2]: Circular tbrook_utility.o <- time_step_module.o dependency dropped. make[2]: Circular tbrook_utility.o <- turbulence_module.o dependency dropped. make[2]: Circular do_diagnostics.o <- time_step_module.o dependency dropped.

這**算是一種警告（Warning），而不是讓編譯立刻中斷的致命錯誤（Fatal Error）**，但它隱藏了程式碼結構上的嚴重問題。

這是什麼意思？

- `GNU Make` 發現了**循環相依（Circular Dependency）**。
- 例如：A 檔案的編譯需要 B 檔案，但 B 檔案的編譯又需要 A 檔案（A \(\rightarrow \) B \(\rightarrow \) A）。
- 為了避免陷入無窮迴圈，`Make` 自動幫你**切斷了其中一條相依鏈**（dependency dropped），並嘗試繼續編譯。

潛在的風險

雖然程式可能成功編譯出執行檔，但會帶來以下代價：

1. **編譯順序錯亂**：`Make` 刪除相依性是隨機的，這可能導致某些模組在它需要的基礎模組編譯完成**之前**就先編譯，進而引發後續的語法或連結錯誤。
2. **重複編譯失效**：當你修改了某個原始碼，`Make` 可能無法正確判斷哪些檔案需要重新編譯，導致你必須每次都執行 `make clean` 才能確保程式碼更新。
3. **架構設計不良**：這通常代表 Fortran 模組（`.mod`）之間發生了循環引用（例如 `module A` 裡使用了 `use B`，而 `module B` 裡又用了 `use A`）。

如何解決？

1. **重構程式碼**：檢查訊息中提到的模組（如 `time_step_module` 和 `diagnostics_module`），將它們共同需要的變數或函式獨立出來，單獨做成第三個模組（模組 C），讓 A 和 B 都去 `use C`。
2. **檢查 Makefile**：確認 Makefile 中的句尾沒有寫錯相依性規則，特別是手動編寫的 Fortran 模組相依鏈。

這背後有兩個主要原因：

原因一：新加入了「其他模組」，導致 `make` 調整了編譯順序

當你在新程式碼中加入了新檔案、或者修改了其他不相關的模組時，`make` 的內部圖形拓撲排序（Topology Sort）會被重新計算。

- **舊程式碼能過**：是因為在舊的結構下，`make` 自動切斷循環相依時，**幸運地**選擇了一條正確的順序（先編譯了 `time_step_module`）。
- **新程式碼失敗**：因為新檔案的加入，干擾了多核心平行編譯（`make -j`）的搶占順序，導致這次 `make` **不幸地**先把 `diagnostics_module` 丟給其中一個 CPU 核心去編譯，而當時 `time_step_module` 還在排隊，於是就爆炸了。

原因二：隱性相依（Implicit Dependency）被觸發了

這兩個模組雖然你沒改，但它們可能透過「第三者」產生了間接關聯。例如：

1. 你可能修改了 `property_module.f 90` 或 `tensor_module.f 90`。
2. `diagnostics_module` 剛好也 `USE` 了這個被你修改的模組。
3. 這導致 `diagnostics_module` 被迫提早重新編譯，進而提早撞上了它與 `time_step_module` 之間本來就存在的循環相依未爆彈。

有時候舊程式碼留下的 `.mod` 檔案還殘留在編譯目錄中，導致新程式碼誤用了舊的模組檔。請先完整清理：

```
diagnostics_module.o: time_step_module.o property_module.o
```

這裡的時間標記規則在做什麼？

這段程式碼的意思是：因為 `drivers.FEXT` 依賴了一個空目標 `FORCE_REBUILD`，所以**每次只要你輸入 `make`，不管程式碼有沒有改，它都會強制重新編譯 `drivers` 相關的檔案**。

專案原作者這樣寫，是為了確保執行檔裡記錄的「編譯時間戳記」永遠是最新的。但這也帶來了一個巨大的副作用：**它每次都會強迫 `make` 重新洗牌一部分的編譯順序**。這剛好完美的解釋了，為什麼你只要動了 Python 腳本或放了新檔案，這裡就會被觸發，然後引爆 `diagnostics_module` 的相依性地獄。

在 Linux 軟體開發的領域中，**「平行編譯」與「單核心編譯」永遠不會影響執行檔是否能進行平行模擬**。

很多人容易把這兩個概念混淆，但它們在作業系統底層代表完全不同的兩件事：

- **編譯階段（Compile-time）**：決定的是「**有沒有把平行模擬的零件（如 MPI）安裝進去**」。
- **執行階段（Runtime）**：決定的是「**要用多少個核心去跑這台裝好平行零件的機器**」。

不論你是用 1 個核心還是 64 個核心去編譯程式，只要編譯時的設定變數（如 `PARALLEL=yes`）與編譯器設定相同，產出的二進位執行檔（Binary）內容就是 100% 一模一樣的。

不過，確實有**少數特定的錯誤操作情況**，會讓你在主觀感覺上認為「因為換了編譯方式，導致無法平行模擬」，主要有以下三種情況：

---

情況一：修改編譯核心數時，不小心改錯了變數或目標（Target）

這是最常見的誤操作。

- **錯誤示範**：  
    在頂層 `Makefile` 裡，平行模擬版本對應的目標是 `all-parallel-opt`，單機序列版是 `all-serial-opt`。  
    如果你為了改成單核編譯，誤把 `./Parallel.sh` 裡面的 `all-parallel-opt` 改成了 `all-serial-opt`，這時編譯器就會**完全拔除 MPI 平行程式碼**。
- **結果**：產出的執行檔變成純單機版，後續就無法使用 `mpirun` 進行平行模擬。
- **正確做法**：目標維持 `all-parallel-opt` 不變，只修改控制編譯執行緒的變數（如 `BuildCPUs=1`）。

情況二：切換編譯方式前，沒有徹底執行 `make clean`

當你從原本失敗的平行編譯（4 核）切換到單核編譯（1 核）時，如果沒有手動清理舊檔案：

- **發生的問題**：`make` 可能會殘留一部分之前平行編譯到一半、結構毀損或連結錯誤的舊 `.o` 檔或舊函式庫（如 `libpgslib.a` 壞掉）。此時直接用單核繼續編譯，雖然順利過關產出執行檔，但執行檔內部的平行模組其實是壞的。
- **結果**：後續平行模擬時會直接閃退或報錯（例如 `MPI_Init` 失敗）。
- **正確做法**：只要變更任何編譯參數，第一步絕對是先執行 `rm -rf build` 或 `make clean`。

情況三：在不支援 MPI 的單機環境上編譯（編譯環境與執行環境不一致）

如果你是在自己的個人筆記型電腦（沒有安裝 MPI 叢集環境）上進行單核編譯，然後把編譯好的執行檔直接複製到高速計算中心（Cluster）的節點上跑平行模擬：

- **發生的問題**：編譯器在單機上找不到平行計算專用的函式庫（如 `libmpi.so`、`CHACO` 等），導致編譯時自動降級退回單機序列版。
- **結果**：到了計算節點上用 `mpirun` 啟動時，它只會變成「多個重複的單機程式在各自算各自的」，而不是互相協同的平行模擬。
- **正確做法**：必須在要執行模擬的計算節點環境（例如你的 ` slider01@waves `）上直接進行編譯。
